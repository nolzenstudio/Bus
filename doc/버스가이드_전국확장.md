## 2. 전국 대체 데이터소스 — 국토부 TAGO

| 지금 (제주 BIS) | 전국 대체 (TAGO) |
| --- | --- |
| 내장 CSV 4,079곳 | `getCrdntPrxmtSttnList` (좌표기반 근접정류소) |
| `searchRouteByStationList.do` | 정류소별경유노선 목록조회 |
| `searchBusRouteStationList.do` | 노선별경유정류소 목록조회 |
| `searchArrivalInfoList.do` | `getSttnAcctoArvlPrearngeInfoList` |

- 엔드포인트 베이스: `http://apis.data.go.kr/1613000/...`
- 인증: `serviceKey` (공공데이터포털 발급, 무료·자동승인)

### 2.1 좌표기반 근접정류소가 두 문제를 동시에 푼다

요청 파라미터가 `serviceKey / gpsLati / gpsLong`뿐이고 **`cityCode`가 필요 없다.** 응답에 `nodeid, nodenm, gpslati, gpslong, citycode`가 온다.

- **정류장 데이터 내장 불가 문제 해결** — 전국 10만+ 정류장을 HTML에 인라인할 방법이 없다. 조회로 대체.
- **도시코드 라우팅 자동 해결** — 좌표 → citycode → 이후 모든 호출에 그대로 사용.
- 부수 효과: 현재 정류장 데이터가 2022-11-01 스냅샷이라 신설/폐지가 반영 안 되는 [알려진 한계]도 같이 사라진다.

**단, 반경 500m 고정.** 지금 UX는 "거리 무관 가까운 8곳"이라 시골에서 결과 0이 나온다.
→ 폴백: 국토부 *전국 버스정류장 위치정보*(파일데이터)를 격자 타일로 쪼개 필요한 구역만 지연 로딩.

### 2.2 응답 필드가 빈다 — 실질적 기능 손실

TAGO 도착정보 응답: `routeid, routeno, routetp, arrtime`(초), `arrprevstationcnt`, `nodenm`, `nodeid`.

| 잃는 것 | 쓰이는 곳 | 대체안 |
| --- | --- | --- |
| **종점명 `DST_NM`** | 노선 그룹화 키 `ROUTE_NUM+DST_NM` (`index.html:1363`), 방면 라벨 전반 | **경유정류소 목록에서 유도.** 순환 노선용으로 이미 구현된 `loopDirName`(`:1185~`)을 전 노선으로 확대. 또는 노선정보항목조회로 `startnodenm/endnodenm` 보강(호출 1회 추가, 캐시) |
| **현재 통과지점 `CURR_STATION_NM`** | 노선 카드 "현재 ○○ 통과" | "n개 정류장 남음"(`arrprevstationcnt`)으로 축소 표기, 또는 버스위치정보 API 추가 |
| **저상버스 `LOW_PLATE_TP`** | 전광판 카드(`:1040`) | `routetp`로 부분 대체 or 표기 제거 |

> 이 때문에 **제주는 기존 어댑터를 유지**하는 편이 낫다. 제주 BIS 쪽 정보량이 TAGO보다 많다.

### 2.3 커버리지 — 가장 큰 미확인 리스크

- TAGO 문서는 "제공 도시는 **도시코드 목록 조회**로 확인"이라고만 하고 목록을 공개하지 않는다.
- 서울(TOPIS)·경기(GBIS)·인천은 자체 시스템이 별도로 있고, TAGO 반영도는 지역마다 다르다는 보고가 많다.
- 마을버스·농어촌버스 누락 편차도 크다.
- **→ 착수 전 첫 작업은 도시코드 목록을 실제로 뽑아 대표 도시 몇 곳을 실측하는 것.** 여기 결과에 따라 범위가 결정된다.

## 3. 호출량 — 설계를 바꿔야 하는 지점

- 개발계정 **일 10,000회**, 운영계정은 활용사례 등록 시 상향 신청 가능.
- 현재는 **정류장당 20초 폴링**(`refreshArrivals`, `:826`).
- 한 사용자가 30분 사용 = 약 90회. **사용자 30명이면 하루치 소진.**

필수 조치:

1. Workers에 **정류장별 15초 공유 캐시** — 같은 정류장을 여러 명이 봐도 상류 호출 1회.
2. ~~**비활성 탭 폴링 중지**(`visibilitychange`)~~ — **2026-07-24 적용 완료**. 숨으면 정지, 복귀 시 즉시 1회 갱신 후 재개(`index.html:823~`).
3. **적응형 폴링 + 로컬 카운트다운** → 아래 3.1.
4. 운영계정 트래픽 상향 신청.

프록시의 역할이 바뀐다: 제주에선 **CORS 우회**용이었지만, TAGO에선 **serviceKey 은닉 + 캐시/쿼터 방어**가 본체가 된다.

### 3.1 적응형 폴링 + 로컬 카운트다운 — ④단계에서 착수

**착수 시점을 ④로 미룬 이유**(2026-07-24 결정): 제주 API는 쿼터가 없어 지금 넣어봐야 배터리 절약뿐인데,
표시 정확도는 미세하게 깎인다. TAGO는 `arrtime`이 **초 단위**라 카운트다운 품질이 제대로 나오고,
그때는 쿼터가 실제 제약이므로 이득이 명확하다. ②(어댑터) 이전에 렌더링을 건드리면 회귀 판단도 어려워진다.

**전제 — 데이터 granularity**

- 제주 `PREDICT_TRAV_TM` = **분 단위 정수**. 20초 폴링의 2/3는 같은 숫자를 다시 받는 낭비.
- TAGO `arrtime` = **초 단위**. 카운트다운의 정당한 근거가 여기서 생긴다.
- **분 단위 원본에서 초 단위 표시를 만들지 말 것** — "5분"은 실제로 5:00~5:59 중 어디든이라, 없는 정밀도를 지어내는 것이 된다.

**로컬 카운트다운이 대체하지 못하는 것 (폴링을 없앨 수 없는 이유)**

1. **ETA 재계산** — 실제 도착정보는 정체·신호로 **늘어나기도** 한다. 단조 감소하는 카운트다운은 항상 낙관적이고, 그건 **버스를 놓치는 방향의 오차**다. 가장 큰 리스크.
2. **버스의 등장/결행** — 새로 접근하는 버스나 사라진 버스는 만들어낼 수 없다.
3. **곁다리 숫자와의 불일치** — 카드는 `5분 · 3정거장`(`etaHTML`), 전광판은 `N개 정류장 남음`을 함께 찍는다. 분만 로컬로 줄면 **"0분 · 3정거장 남음"** 같은 모순이 난다.

**설계 — 균일하게 늘리지 않고, 임박할수록 촘촘히**

| 가장 빠른 버스 | 폴링 주기 | 근거 |
| --- | --- | --- |
| ≤ 3분 | **20초**(현행 유지) | 탈지 말지 결정하는 구간, 오차 허용 안 됨 |
| 4~10분 | **60초** | 분 단위 데이터엔 충분 |
| 11분+ / 접근 버스 없음 | **120초** | 어차피 화면을 안 보고 있음 |

안전장치:

- **로컬 차감이 서버값보다 2분 앞서면 즉시 재조회** — 카운트다운이 현실과 벌어지는 것을 스스로 감지
- **1분에서 멈추고 "곧 도착"으로 전환** — 0분을 로컬로 만들지 않는다(전광판은 이미 1분 이하에 이 문구를 쓴다)
- 로컬 차감된 값은 `약 4분`처럼 **완화 표기**해 서버 확인값과 구분. 정거장 수는 차감 대상이 아니므로 손대지 않는다

**절감 효과 (30분 세션 기준)**

| 방식 | 호출 수 |
| --- | --- |
| 현행 20초 고정 | 90회 |
| 60초 고정 | 30회 |
| **적응형** | 대략 25~35회 (임박 구간에만 촘촘) |

60초 고정과 절감폭은 비슷한데 **임박 구간 정확도는 그대로** 지킨다. 여기에 Workers 공유 캐시가 붙으면 같은 정류장을 보는 사용자가 몇이든 상류 호출은 하나다.

## 4. 구조 제안 — provider 어댑터

메서드 4개짜리 인터페이스 하나만 맞추면 렌더링 코드는 손대지 않아도 된다.

```js
// 어댑터는 반드시 '제주 필드명'으로 정규화해서 반환한다.
// (STATION_ID / STATION_NM / STATION_ORD / ROUTE_ID / ROUTE_NUM / DST_NM /
//  PREDICT_TRAV_TM / REMAIN_STATION / CURR_STATION_NM)
const provider = {
  nearbyStations(lat, lng),   // → [{STATION_ID, STATION_NM, lat, lng, dist}]
  stationRoutes(stationId),   // → [{ROUTE_ID, ROUTE_NUM, DST_NM}]
  routeStations(routeId),     // → [{STATION_ID, STATION_NM, STATION_ORD}]
  arrivals(stationId),        // → [{ROUTE_ID, ROUTE_NUM, PREDICT_TRAV_TM, REMAIN_STATION, CURR_STATION_NM}]
};
```

- 좌표로 지역을 판정해 어댑터 선택: `jeju`(기존 유지) / `tago`(기본) / 향후 `seoul`·`gbis`
- 기존 3개 캐시(`fetchStationRoutes` / `getRouteStations` / `findDirectBetween`)는 **어댑터 위에 그대로** 둔다. 우회하면 중복 요청이 나는 규칙은 변하지 않는다.
- 폴러 일원화 규칙(정류장당 `refreshArrivals` 하나)도 유지.

### 그대로 가는 것들

- 2단 중첩 아코디언, 패널 슬롯↔보관함 이동 방식, 카드 안쪽만 교체하는 재렌더 전략
- 목적지 필터의 카카오 장소검색 (전국 그대로 동작 — `inJejuBounds`만 지역화)
- 환승 미지원(직통만) 정책
- 랜드마크 기반 방면 라벨 로직 — 표만 소폭 보강

## 5. 작업 순서

| 단계 | 내용 | 규모 |
| --- | --- | --- |
| ① | serviceKey 발급 → **도시코드 목록 실측**, 대표 도시(대전·부산·광주 등) 응답 품질 확인. 서울/경기 포함 여부 확정 | 반나절 |
| ② | **provider 어댑터 리팩터링** — 제주 어댑터로 감싸고 기존 동작 회귀 확인. *확장을 안 하더라도 이 단계는 이득* | 하루 |
| ③ | 폴링/캐시 재설계 (Workers 공유 캐시 + ~~`document.hidden` 가드~~ **2026-07-24 완료**) | 반나절 |
| ④ | **TAGO 어댑터 + 한 도시 파일럿** — 방면 유도 로직 확대, 근접정류소 폴백, **적응형 폴링+로컬 카운트다운(3.1)** | 하루 |
| ⑤ | 제주 특화 UX 일반화, 지역별 품질 튜닝, 전국 오픈 | 미정(대부분의 시간) |

코드 작업은 ①~④ 합쳐 이틀 남짓. **⑤가 실제 비용**이다.
